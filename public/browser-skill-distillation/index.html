Method

The trace-to-skill pipeline

BrowserBC lifts behavior cloning from one-step action imitation to graph-grounded skill distillation. The page below turns the method section into a compact flow that is easier to scan than the full PDF.

1

Clean trajectories

Normalize heterogeneous logs and remove interactions that do not change the task state.

2

Build graphs

Convert demonstrations into state-transition graphs with progress, verification, recovery, and failure edges.

3

Induce skills

Cluster reusable subgraphs into candidate procedures supported by multiple trajectory fragments.

4

Distill cards

Summarize each skill into intent, scope, observations, steps, evidence, failure modes, and provenance.

5

Retrieve at runtime

Rank skills by relevance, compatibility, graph connectivity, and historical reliability before execution.

OSWorld transfer

Skills are procedural priors, not browser-only tricks.

The updated paper adds an OSWorld-style desktop case study. It does not claim a full OSWorld leaderboard result; instead, it tests whether the same skill abstraction remains useful when the interface changes from web pages to Ubuntu desktop applications, files, dialogs, and persistent state.

What transfers is the state-transition model: preconditions, progress milestones, terminal evidence, and recovery strategies. The low-level GUI control problem remains separate.

30 desktop cases plus controls

Counts diagnose when skills help, are unnecessary, fail, or mislead.

Procedural uncertainty17

The base agent reaches a relevant surface, but the skill supplies the missing schema and terminal evidence.

Low uncertainty2

The task is shallow enough that the base agent already finds the action path.

Execution fragility10

The procedure may be correct, but focus, modals, file pickers, terminal robustness, or office recovery break execution.

Poor calibration3

A plausible but wrong skill can confidently move the agent away from the target state.

What OSWorld clarifies

Skills help When the missing ingredient is workflow structure: what to open, edit, persist, and verify.
Skills are limited When the bottleneck is low-level GUI grounding, hidden focus state, or application recovery.
Skills need calibration Retrieved procedures should be checked against live state instead of followed as commands.
Skill card anatomy

What gets stored in the skill library?

A distilled skill is meant to be auditable and executable. It does not store benchmark answers; it stores procedure, evidence, and boundaries that help the agent choose actions on a live page.

Before acting

Intent and preconditions

  • What user goal the skill advances.
  • Which page state or domain it applies to.
  • What observations should be checked first.
During action

Procedure and grounding

  • High-level steps without brittle selectors.
  • Element-selection principles.
  • Milestones that indicate progress.
After action

Evidence and recovery

  • Terminal signs that the subtask is done.
  • Common failure modes.
  • Fallback paths when the page diverges.
The key design choice is confidence-aware use: the skill is a procedural prior, not a command to blindly follow. The paper notes that over-specific skills can occasionally regress tasks, so the agent should defer to live page evidence when it conflicts with the retrieved procedure.
